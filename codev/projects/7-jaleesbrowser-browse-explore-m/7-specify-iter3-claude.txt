### Minor Comments (none blocking)

1. **C2 still listed as "open" but has a firm position.** The spec says `raw-by-SHA, lazy-by-page with concurrent metadata fetch` — this reads like a decision, not an open question. Consider marking it DECIDED like C1 to avoid plan-phase confusion about whether this is still negotiable. (Cosmetic — the position is clear enough.)

2. **Multi-value query encoding within an axis unspecified.** §5.3 shows `?<axis>=<val>` but for OR-within-axis, a user might filter by multiple values (e.g., `pillars=restraint` AND `pillars=patience`). Is this repeated params (`?pillars=restraint&pillars=patience`) or comma-separated (`?pillars=restraint,patience`)? Since the spec says "the query string *is* the state" and deep-linkability is a MUST, the encoding matters for stable URLs. This is arguably a plan/implementation detail, but a one-liner in §5.3 or M5 would remove ambiguity.

3. **Results-ready seam: SHOULD vs. the tone of §4.1.** §7 classifies the results seam as SHOULD (S3), but §4.1 describes it with definitive language ("v1 builds exactly three small, inert seams") and §9.7 tests for it. A builder reading §7 alone might treat S3 as optional. If the seam is expected (which the rest of the spec clearly implies), it could be a MUST. Not blocking — the intent is clear from context, and both prior reviewers treated it as expected.

4. **HTTP status codes for error states.** §5.5 specifies the *rendered behavior* (friendly error page, banner notice, 404 for removed content) but doesn't specify status codes for non-404 errors. E.g., when GitHub is down and there's no cache: is the friendly error page served with 503 or 200? A sentence would help the builder, but this is firmly plan-level.
