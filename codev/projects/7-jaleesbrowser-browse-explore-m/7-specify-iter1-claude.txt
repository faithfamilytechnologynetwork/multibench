## Specification Review: 7-jaleesbrowser — browse & explore MultiBench traditions

### Overall Assessment

This is an exceptionally well-crafted spec. The §2.1 reframing ("this is not JaleesBench's results browser") is the single most valuable sentence in the document — it prevents the dominant over-build failure mode. The requirements are concrete, the scope boundaries are sharp, and the open questions are honestly categorized. I have one substantive concern and several clarifications worth addressing before this goes to human review.

---

### 1. Completeness

**Strong.** Requirements M1–M11 cover the full browsing flow. Success criteria §10 are testable. Out-of-scope §4 is unusually well-drawn.

**Gaps to address:**

**1a. Filter semantics within an axis (M7).** The spec says "Filters are composable (AND across axes)" but is silent on the behavior *within* a single axis. If the `pillars` axis has values `[courage, justice, patience, restraint, cross_cutting]` and a user selects both `courage` and `justice`, is that AND (scenario must have both tags) or OR (scenario has at least one)? Standard tag-filtering UX is OR-within-axis, AND-across-axes. This matters for the builder — it changes the filter logic and the UI affordance (checkboxes vs. radio buttons). **Recommend: state "OR within axis, AND across axes" explicitly in M7, or flag it as a Plan decision.**

**1b. Index ↔ folder divergence behavior (M6).** The spec says the scenario list comes "from `scenarios/index.json`, cross-checked against the `scenarios/*/` folders." In display-first mode, what happens when they diverge? Three cases: (a) folder exists but not in index, (b) index entry but no folder, (c) both present. The validator treats divergence as an error. The browser's display-first posture needs a stated policy — e.g., "show the union with an inline notice on orphans/ghosts." **Recommend: one sentence in M6 or §8 specifying the display-first divergence behavior.**

**1c. Static site link integrity.** §10 mentions unit tests and an integration test, but for an SSG the classic failure mode is broken inter-page links (especially prev/next navigation, tradition→scenario links, and the back-to-index link). **Recommend: add a sentence to §10.7 noting link-integrity verification in the integration test.**

### 2. Correctness

**Verified against the codebase.** Key claims check out:

- ✅ The §2.4 rename table is accurate. Current disk has `probes/`, `probe.yaml`, `scenario.md` (for turn-1), `probes/index.json`, and the manifest uses `probe_id_pattern`. The spec's post-rename targets (`scenarios/`, `scenario.yaml`, `turn1.md`, `scenario_id_pattern`, `scenarios/index.json`) are consistent.
- ✅ The validator's models are indeed `extra="forbid"` + `strict=True` — confirming the spec's correct identification (§8, I1) that direct reuse requires wrapping for tolerant reading.
- ✅ `core.py` constants (PRESSURES, FRAMINGS, IDENTITY_SIGNALS, STATED_TEMPLATE) match the spec's §2.4 universal core description exactly.
- ✅ The validator's `_within_root()` symlink/escape check exists in `validator.py:33-39`, and the loaders enforce size caps and UTF-8 — the spec should note these safety measures carry over or are replicated (see §5 below on security).

**One subtle point on the rename table:** The spec maps `scenario.md` → `turn1.md`. This is the *turn-1 prose file* being renamed, which makes sense (post-rename, `scenarios/JLS-001/scenario.md` would be confusingly redundant). I'm flagging this only because this rename isn't as obviously implied by "#6 probe→scenario rename" as the others — ensure #6 actually renames this file, or the constants module will need to account for it.

### 3. Feasibility

**Approach A is sound.** Verified the ecosystem:

- Jinja2 + `markdown-it-py` + Typer + Pydantic is a well-trodden Python SSG stack. No exotic dependencies.
- 140 scenarios × a handful of traditions → ~150-200 HTML pages per tradition. Build time will be sub-second. The "no noticeable lag" non-functional requirement is trivially met.
- The `serve --watch` convenience: the spec uses this as a key argument for A over B. Note that Python stdlib has no filesystem watcher — this requires `watchfiles` or polling. Not a blocker, but the Plan should pick a mechanism. The spec correctly defers to Plan.

**Dependency topology concern (I1).** If jaleesbrowser imports from `tradition_validator`, it needs a `uv` path dependency (`tradition_validator = {path = "../tradition_validator"}`). This works but creates a build-order coupling between two independent apps. The alternative — vendoring a slim read-model that shares the *shape* but not the *strictness* — is cleaner for an independent app. The spec correctly defers this to Plan, but the feasibility of both paths is confirmed.

---

### 4. Security

**Adequate but could be more explicit.**

- ✅ Markdown sanitization is mentioned (§2.5, I3) — defense-in-depth against semi-trusted repo content.
- ✅ The validator has symlink/path-traversal rejection (`_within_root()`).

**Recommend adding one sentence to §3 or §8:** "jaleesbrowser replicates the validator's path-containment check (no symlinks or `..` escaping the tradition root) and the size-cap guard." This makes it explicit that the builder must carry forward these safety properties rather than hoping they come "for free" from model reuse.

Also: the generated static site should not reference external CDNs (CSS/JS), which would break the zero-install/offline philosophy and introduce a supply-chain surface. The "no network calls" non-functional requirement in §10 covers runtime, but the *output artifact* should also be self-contained. **Recommend: add "generated site is fully self-contained (no external CDN references)" to the non-functional requirements in §10.**

---

### 5. Clarity

**Excellent.** The spec reads like it was written for a builder who has never seen the codebase. §2.1 is a model for how to prevent misunderstanding. §2.4's data-format table is precise. The M/S/C prioritization is clean.

**One terminology note:** The spec uses "inline notice" (§7.3 M8, §8) for display-first degradation but never defines what that means concretely. Is it a visible banner/badge in the rendered HTML? A CSS-styled warning block? A `<details>` element? The Plan will need to decide, but the spec should at least say "a visually distinct warning rendered in the page" so the builder doesn't interpret "notice" as a console log.

### 6. Open Questions Assessment

The open questions are well-triaged. My takes on the critical ones:

- **C1 (A vs B):** A is the right call. B's "always live" advantage is real but the `serve --watch` workflow closes 90% of the gap, and A's static artifact is genuinely more useful (hostable, shareable, diffable). The consultation should pressure-test this, as the spec requests.
- **C2 (#6 sequencing):** Building against post-rename with synthetic fixtures is correct. The constants-module isolation makes the #6 rebase a one-file edit. Pragmatic and low-risk.
- **I1 (model reuse):** The validator's `extra="forbid"` + `strict=True` means direct import will *reject* any scenario with an unexpected field (e.g., a future `notes` field an author adds during drafting). Display-first requires either wrapping with `try/except` per-model or a separate lenient model. The spec correctly identifies this tension.
