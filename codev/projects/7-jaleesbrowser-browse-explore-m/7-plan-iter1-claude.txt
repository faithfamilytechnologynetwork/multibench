### 2. Phase Breakdown — Clean and Logical

The strictly linear dependency chain (`P1 → P2 → P3 → P4 → P5`) is the right structure for this project. Each phase is independently committable and testable — a builder can run the test suite after each phase and get a green bar.

**Sizing is well-balanced:**
- P1 (scaffold + safeio + discovery): foundational, right-sized for a first commit
- P2 (complete data model): the meaty data layer — appropriately isolated from rendering
- P3 (templates + markdown): the rendering layer as pure functions (HTML strings, no I/O)
- P4 (build + client JS): the integration phase — largest but unavoidably so
- P5 (serve + polish): light finishing

The data/presentation split (P1–P2 vs P3–P5) is a strong architectural choice — it means the data layer is fully testable before any HTML exists, and a bad template can't block data-layer work.

### 3. Technical Approach — Sound

**I1 decision (vendor, don't import):** Verified against the codebase — `tradition_validator` models use `ConfigDict(extra="forbid", strict=True)`, which would reject unknown keys that a display-first browser should render with a notice. The plan's dataclass-based read-model with tolerant parsing is the right call. Re-implementing `_within_root` and `MAX_FILE_BYTES` locally is a trivial cost for clean decoupling.

**I3 decision (markdown-it-py + nh3):** Good picks. `markdown-it-py` is the Python port of the JS `markdown-it` (same family as the reference), actively maintained, Unicode-correct. `nh3` is the Rust-based successor to the deprecated `bleach`. The "raw HTML disabled + sanitizer" belt-and-suspenders approach matches §8's security requirements.

**Constants isolation:** The plan puts all post-rename names in `constants.py` — verified that the current codebase uses `probe.yaml` / `scenario.md` / `probe_id_pattern` / `probes/`, so at rebase time, only `constants.py` needs updating. Smart.

**Mirroring tradition_validator conventions:** Confirmed the plan's target layout matches the sibling app's actual structure (`__main__.py` → `cli.py:app`, Typer, flat package, hatchling build backend).

### 5. Risk Assessment — Well-Calibrated

The #6 dependency is the top risk and is handled thoroughly (synthetic fixtures, constants isolation, rebase-in-Verify, explicit Validation Checkpoint #4). The probability/impact ratings are realistic.

**One risk I'd add (minor):** The `watchfiles` package is Rust-backed, which means it needs a Rust compiler or a pre-built wheel at install time. On standard platforms this is fine (wheels exist), but it could surprise a builder on an unusual arch. Since `watchfiles` is only needed for `--watch` (not the core `build`/`serve`), this is low-impact. The plan already notes keeping `--watch` untested-in-CI.

### 6. Minor Observations (non-blocking)

1. **Phase 4 is the densest phase** — it includes build orchestration, client-side JS, CLI `build` command, filter-index JSON, determinism guarantee, link-integrity check, AND read-only snapshot invariant. A builder could benefit from a suggested ordering within the phase (e.g., site.py output paths first → build orchestration → filter-index → filter.js → CLI wiring → tests). 3. **The plan's constants.py lists `normalize_heading()`** as a function — it's good that heading normalization is centralized alongside the format constants, since it's the bridge between raw `## ` headings and the canonical pressure keys.
