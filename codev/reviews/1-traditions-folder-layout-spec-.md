## 1. What was built

A **canonical, file-based, human-and-machine-readable format** for a MultiBench
*tradition*, a **validator** that enforces it, the **Sunni Islam** tradition ported into
that format, full documentation, and a **`create-tradition` skill**.

- **Format** — a self-contained `traditions/<id>/` directory: `tradition.yaml` (manifest)
  + prose Markdown (`README.md`, `source.md`, `guide.md`) + one folder per probe
  (`probe.yaml` + `scenario.md` + `judge-guidance.md` + `pressures.md`) + a tiny
  `probes/index.json`. **No large JSON.** Framings/pressures are universal core; the only
  faith-specific framing inputs are `adherent_noun`, `guide.md`, and per-probe pressures.
- **Validator** — `apps/tradition_validator/` (Python/uv/Typer/Pydantic v2/PyYAML).
  Closed + strict schemas, the judge-guidance "seam", `pressures.md` heading coverage,
  index⟺folders drift, prose non-emptiness, and safety guards (UTF-8, YAML safe-load,
  symlink-escape rejection, size cap). `validate` / `validate-all`, `--strict`,
  `--format text|json`.
- **Sunni Islam port** — all **140** JaleesBench probes, generated by `port_jaleesbench.py`
  (ast-parses the source, no exec) and validating **clean** (the acceptance test).
- **Docs** — `traditions/README.md` (the format doc), the validator README, `apps/README.md`,
  refreshed root README.
- **Skill** — `.claude/skills/create-tradition/SKILL.md`.

## 3. What went well
- **Deriving the format from real JaleesBench data** (not docs) up front — the proof-text
  *seam* insight (judge anchors to the embedded text, not a corpus lookup) shaped the whole
  design and was praised by all reviewers across iterations.
- **Test-first against fixtures, then the real 140-probe port as the acceptance test** —
  the port surfaced no format gaps; the format genuinely expresses a real tradition.
- **Closed + strict Pydantic schemas** gave precise, located errors almost for free and
  satisfied the unknown-key and no-coercion requirements.

## 4. What was challenging
- **Porch checks were Node defaults** in a Python repo (`npm run build`/`npm test`) — they
  hard-block `porch done`. Resolved via `.codev/config.json` `porch.checks` (config >
  protocol).
- **Gemini's per-phase impl consult can't see the worktree** (empty sandbox / permission
  stall → no verdict → parsed REQUEST_CHANGES), which would force every phase to the
  iteration ceiling. Resolved via `porch.consultation.models: ["codex","claude"]`; the
  architect runs a full 3-way integration CMAP at the PR gate (diff fed inline).
- **Adding Phase 6 mid-implement** required a `porch rollback 1 plan` + re-approval because
  porch only re-extracts plan phases at the plan→implement transition.

## 5. Systematic issues / methodology notes
- Both config issues (npm checks, blind Gemini) are **environment/protocol-default**
  mismatches for a Python project, not code problems — worth a project-level note so the
  next Python builder doesn't rediscover them (captured below).
- Consult flakiness: a Codex run occasionally fails to write its output (porch's "run
  remaining consultations" handles it — just re-run that model).

## Architecture Updates
*(Routed to the hot-tier `codev/resources/arch-critical.md`; cold detail in `arch.md`.)*
- A **tradition** is a drop-in `traditions/<id>/` directory in the canonical **file-based**
  format (prose=Markdown, metadata=small YAML, only `probes/index.json` is JSON); discovery
  globs `traditions/*/tradition.yaml`. Contract: Spec 1 / `traditions/README.md`.
- The **judge seam**: each probe's `judge-guidance.md` *is* the judge's binding ground
  truth — there is no separate proof-text corpus. Don't reintroduce one.
- **Framings (unstated/stated/guided) and the six pressures are universal core**, identical
  across traditions; per-tradition only `adherent_noun`, `guide.md`, per-probe `pressures.md`.
- The **`tradition_validator`** (`apps/tradition_validator/`) is the mechanical gate for any
  tradition before workflows consume it; run it from the repo root via `uv --project`.

## Lessons Learned Updates
*(Routed to the hot-tier `codev/resources/lessons-critical.md`; cold detail in `lessons-learned.md`.)*
- This is a **Python (uv) repo**: porch's default implement-phase checks are npm and hard-
  block; set `.codev/config.json` `porch.checks` (build skip, tests = `uv run pytest` with
  `cwd`). Already configured.
- **Gemini's per-phase impl/code consult can't see the worktree** here → it returns no
  verdict (blocks unanimity). Per-phase consultation is set to `["codex","claude"]`; do the
  full 3-way only where the diff is fed inline (the PR integration CMAP).
- Derive a data format from the **real reference data**, not its docs — the load-bearing
  details (e.g. the embedded judge anchor) only show up there.

## Flaky tests
None. (One Codex *consult* invocation failed to write output once — a consult-tooling
hiccup, not a test; re-running the single model resolved it.)

## Methodology / protocol suggestions
- The SPIR skeleton's implement/review checks assume a Node project; a Python-project
  preset (or auto-detection) would save the per-project `.codev/config.json` override.
- Document the post-rebuttal porch cadence (it can take two `porch done`s — one to record
  the prior iteration + bump, one to mark the new iteration's build complete — before
  `porch next` emits the consult).
